# Техническое задание

## 1. Общие сведения

**Наименование ПО:** программное обеспечение блока управления камерой создания пылевоздушных смесей.

**Назначение ПО:** управление узлами камеры создания пылевоздушных смесей, ручная подача пыли оператором, отображение текущих параметров работы установки, отображение показаний эталонного прибора, журналирование действий оператора, обеспечение локальной и удаленной аварийной остановки установки.

**Платформа:** Raspberry Pi.

**Язык разработки:** Python.

**Среда эксплуатации:** Raspberry Pi под управлением Linux.

## 2. Цель разработки

Разработать ПО для блока управления установкой, обеспечивающее:

- ручное управление подачей пыли в колонну;
- запуск и остановку рабочих режимов;
- настройку параметров впрыска;
- настройку и контроль параметров давления;
- отображение показаний эталонного прибора;
- отображение состояния исполнительных устройств;
- ведение журнала событий и действий оператора;
- поддержку удаленного доступа;
- поддержку удаленной аварийной остановки установки.

## 3. Функциональное назначение ПО

ПО должно обеспечивать:

- управление исполнительными устройствами установки;
- ручную подачу пыли по команде оператора;
- настройку параметров впрыска;
- настройку допустимых пределов давления;
- отображение текущих данных с датчиков;
- отображение данных эталонного прибора;
- отображение текущего состояния установки;
- журналирование действий оператора и событий системы;
- сохранение настроек и журнала работы;
- перевод системы в аварийный режим при локальной или удаленной команде аварийной остановки;
- удаленный доступ к функциям мониторинга и аварийной остановки через подключение по мобильной сети.

## 4. Состав автоматизируемых функций

### 4.1. Управление установкой

ПО должно обеспечивать:

- запуск системы;
- остановку системы;
- ручную подачу пыли;
- блокировку подачи пыли при аварийном режиме;
- переход в безопасное состояние при аварии.

### 4.2. Управление параметрами впрыска

Оператор должен иметь возможность задавать:

- длительность впрыска;
- интервал между впрысками;
- количество впрысков;
- длительность цикла.

### 4.3. Управление параметрами давления

Оператор должен иметь возможность задавать:

- нижний допустимый предел давления;
- верхний допустимый предел давления.

ПО должно обеспечивать:

- чтение текущих значений давления;
- сравнение текущих значений с установленными пределами;
- блокировку отдельных действий при недопустимых значениях давления.

### 4.4. Работа с эталонным прибором

ПО должно обеспечивать:

- получение текущих показаний эталонного прибора;
- отображение показаний на экране;
- сохранение показаний в журнале.

### 4.5. Журналирование

ПО должно вести журнал:

- запуска и остановки системы;
- подачи пыли;
- изменения настроек;
- срабатывания аварийной остановки;
- ошибок и предупреждений;
- текущих показаний эталонного прибора и датчиков в моменты значимых событий.

### 4.6. Удаленный доступ

ПО должно обеспечивать защищенный удаленный доступ через сеть мобильного оператора для:

- просмотра текущего состояния установки;
- выполнения удаленной аварийной остановки.

### 4.7. Аварийная остановка

ПО должно поддерживать:

- локальную аварийную остановку;
- удаленную аварийную остановку.

При аварийной остановке ПО должно:

- немедленно запретить подачу пыли;
- закрыть исполнительный клапан впрыска;
- остановить компрессор или выдать команду на его остановку;
- перевести систему в состояние `EMERGENCY`;
- зафиксировать событие в журнале;
- запретить повторный запуск до ручного сброса аварийного состояния оператором.

## 5. Требования к интерфейсу пользователя

ПО должно иметь графический интерфейс пользователя.

### 5.1. Главный экран

На главном экране должны отображаться:

- кнопка «Пуск»;
- кнопка «Стоп»;
- кнопка «Аварийная остановка»;
- кнопка «Подать пыль»;
- текущий статус установки;
- текущее давление;
- текущее состояние исполнительных устройств;
- текущие показания эталонного прибора;
- индикация аварийного состояния при наличии.

### 5.2. Экран настройки впрыска

Экран должен содержать поля:

- длительность впрыска;
- интервал между впрысками;
- количество впрысков;
- длительность цикла;
- кнопка сохранения настроек.

### 5.3. Экран настройки давления

Экран должен содержать:

- нижний предел давления;
- верхний предел давления;
- текущее значение давления;
- кнопку применения настроек.

### 5.4. Экран журнала

Экран должен обеспечивать просмотр:

- времени события;
- типа события;
- описания события;
- состояния системы на момент события.

### 5.5. Экран информации о системе

Экран должен отображать:

- версию ПО;
- статус подключения устройств;
- статус удаленного подключения;
- служебную информацию о системе.

## 6. Логика работы системы

ПО должно поддерживать состояния:

- `IDLE` — ожидание;
- `READY` — готовность к работе;
- `RUNNING` — рабочий режим;
- `INJECTION` — выполнение подачи пыли;
- `STOPPED` — остановка;
- `EMERGENCY` — аварийный режим;
- `FAULT` — ошибка оборудования или связи.

### 6.1. Ограничения переходов

- Подача пыли разрешена только при допустимом давлении и отсутствии аварии.
- При состоянии `EMERGENCY` все рабочие операции должны быть заблокированы.
- Выход из состояния `EMERGENCY` допускается только после ручного подтверждения оператором.

## 7. Требования к безопасности

- ПО не должно допускать подачу пыли при недопустимом давлении.
- ПО должно гарантировать закрытие клапана впрыска при любой ошибке выполнения операции.
- При потере связи с датчиком или исполнительным устройством система должна перейти в безопасное состояние или заблокировать опасные действия.
- Удаленная аварийная остановка является дополнительной функцией и не заменяет локальную аппаратную аварийную остановку.
- Потеря удаленной связи не должна приводить к неконтролируемой работе установки.
- Все аварийные события должны сохраняться в журнале.

## 8. Требования к программной архитектуре

ПО должно быть реализовано модульно.

Рекомендуемое логическое разделение:

- модуль пользовательского интерфейса;
- модуль бизнес-логики управления;
- модуль работы с исполнительными устройствами;
- модуль работы с датчиками;
- модуль работы с эталонным прибором;
- модуль журналирования;
- модуль хранения настроек;
- модуль удаленного доступа;
- модуль аварийной остановки.

Архитектура должна обеспечивать возможность замены конкретной реализации оборудования без переработки всей системы.

## 9. Требования к взаимодействию с оборудованием

ПО должно поддерживать взаимодействие с:

- исполнительным устройством подачи пыли;
- компрессором;
- датчиками давления;
- эталонным прибором.

Конкретный способ подключения определяется аппаратной частью проекта и может быть реализован через:

- GPIO;
- UART / RS-485;
- USB;
- Ethernet;
- иные интерфейсы, используемые в составе установки.

Интерфейсы взаимодействия должны быть абстрагированы на уровне программной архитектуры.

## 10. Требования к хранению данных

ПО должно обеспечивать хранение:

- пользовательских настроек;
- параметров впрыска;
- параметров давления;
- журнала событий;
- журнала аварий;
- технических логов.

Настройки должны сохраняться между перезапусками системы. Журнал должен быть доступен для просмотра оператором.

## 11. Требования к удаленному доступу

- ПО должно поддерживать работу через подключение Raspberry Pi к сети мобильного оператора.
- Удаленный доступ должен использовать защищенный канал связи.
- Удаленно должны быть доступны просмотр текущего состояния установки и выполнение аварийной остановки.
- Все удаленные действия должны журналироваться с указанием времени и типа действия.
- Доступ к удаленным функциям должен быть ограничен авторизованным пользователям.
- SIM-карта и мобильный интернет относятся к аппаратной/сетевой инфраструктуре Raspberry Pi. В текущей версии ПО предполагает, что подключение к сети уже настроено на уровне операционной системы через LTE-модем, LTE HAT, USB-модем или внешний роутер.
- ПО должно предоставлять программные функции удаленного мониторинга и аварийной остановки поверх уже доступного сетевого подключения.

## 12. Требования к надежности

- ПО должно автоматически запускаться после включения Raspberry Pi.
- ПО должно корректно восстанавливать сохраненные настройки после перезапуска.
- При программном сбое система должна стремиться перевести исполнительные устройства в безопасное состояние.
- ПО должно вести технический журнал ошибок.
- Не допускается зависание интерфейса при выполнении операций чтения датчиков и обмена с устройствами.

## 13. Требования к удобству эксплуатации

- Интерфейс должен быть понятен оператору без специальной подготовки в программировании.
- Основные действия должны быть доступны с главного экрана.
- Критические действия должны быть визуально выделены.
- Отображаемая информация должна обновляться в реальном времени либо с минимальной задержкой.
- Сообщения об ошибках должны быть понятны пользователю.

## 14. Требования к тестированию

Должны быть проверены:

- запуск и остановка ПО;
- ручная подача пыли;
- блокировка подачи пыли при аварии;
- корректность переходов между состояниями;
- чтение показаний датчиков;
- чтение показаний эталонного прибора;
- сохранение и загрузка настроек;
- ведение журнала;
- срабатывание локальной аварийной остановки;
- срабатывание удаленной аварийной остановки;
- восстановление после перезапуска.

## 15. Результаты разработки

По завершении работ должны быть переданы:

- исполняемое программное обеспечение для Raspberry Pi;
- исходный код программного обеспечения;
- инструкция по запуску и эксплуатации;
- описание структуры настроек;
- описание способа подключения и настройки удаленного доступа;
- краткое руководство оператора.

## 16. Ограничения текущей версии

В рамках данной разработки реализуется упрощенный вариант ПО:

- без автоматического режима калибровки пылемеров;
- без расчета поправочных коэффициентов;
- без формирования полного метрологического протокола калибровки.
- без настройки SIM-карты, LTE-модема и сетевой инфраструктуры внутри приложения.

Текущая версия ПО включает:

- ручное управление подачей пыли;
- настройку основных параметров;
- отображение текущих данных;
- журналирование;
- локальную и удаленную аварийную остановку.

## 17. Уточненный рабочий сценарий

Предварительный сценарий работы установки:

1. При включении установки запускается компрессор.
2. После подготовки компрессора подключается или открывается баллон/пневмолиния.
3. Оператор нажимает кнопку подачи пыли.
4. ПО выполняет подачу пыли через исполнительный клапан при соблюдении условий безопасности.
5. Подключается эталонный прибор.
6. ПО получает и отображает данные эталонного прибора.

Этот сценарий должен быть уточнен после согласования аппаратной части: что именно означает "баллон" в логике управления, есть ли у него отдельный исполнительный клапан/датчик состояния, и должен ли эталонный прибор подключаться до подачи пыли или после нее.
